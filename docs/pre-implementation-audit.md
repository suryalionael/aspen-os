# Pre-Implementation Audit — Sprint 1

**Reviewers:** Founder/PM, CTO, Product Engineer, UX Reviewer, QA Lead
**Status:** Audit complete — items below require explicit decisions before/during build. This document does not change scope; it surfaces gaps, conflicts, and risks in the existing plan for leadership sign-off.
**Reviewed against:** [`prd-sprint-1.md`](prd-sprint-1.md) · [`architecture.md`](architecture.md) · [`database-schema.md`](database-schema.md) · [`technical-plan.md`](technical-plan.md) · [`ux-review.md`](ux-review.md) · [`executive-summary.md`](executive-summary.md) · [`../ROADMAP.md`](../ROADMAP.md)

Severity scale: **Critical** (blocks Go/No-Go or causes data exposure) · **High** (will visibly break a stated requirement) · **Medium** (will cause rework or confusion if unaddressed) · **Low** (worth a decision, low immediate cost).

## 1. Contradictory Requirements

| # | Issue | Severity | Impact | Recommendation |
|---|---|---|---|---|
| C-1 | **Default Supabase email confirmation breaks AC-6.** `architecture.md` §5 specifies "email/password auth (Supabase Auth default)." Supabase's default project setting requires email confirmation before a user can sign in. `prd-sprint-1.md` AC-6 requires a brand-new user to reach a created first task in **≤ 3 minutes with no guidance**, and `ux-review.md` §2 explicitly designs the onboarding flow assuming no interruption between sign-up and workspace creation. A mandatory "check your email, click the link, come back" step makes AC-6 unmeasurable, and in practice unmeetable for most first-time users. | **Critical** | Sprint Success Metrics (`prd-sprint-1.md` §7) and the Go/No-Go criteria (`executive-summary.md` §9) cannot be honestly evaluated if this is left to Supabase's default. | Founder/PM and CTO must make an explicit decision: disable email confirmation for Sprint 1 (auto-confirm on sign-up) to preserve AC-6, or revise AC-6 to exclude email-confirmation time. Document the choice in `architecture.md` §5 — it is currently silent on this. |
| C-2 | **"Shared with the whole team" vs. no invite mechanism.** `prd-sprint-1.md`'s Problem statement and US-5 frame Aspen OS as giving "the whole team" a shared view, and `workspace_members` plus its RLS policy exist to support multiple members. But `architecture.md` §4 states `workspace_members` INSERT is "restricted to the creation flow (self-membership on workspace creation) — no separate invite flow in Sprint 1." There is no path, anywhere in the plan, for a second person to ever join a workspace. | **High** | As scoped, a workspace can only ever have one member. The core value proposition ("shared... for the whole team") is not achievable by the shipped feature set — this is a direct contradiction between the Problem/User Story framing and the actual RLS/feature scope. | Founder/PM decision needed: either (a) reword US-5 and the Problem statement to be honest about single-member Sprint 1 scope, or (b) explicitly evaluate adding a minimal invite mechanism as in-scope — following the same "evaluate, don't assume" pattern already used for the Org layer and "My Tasks" decisions. Do not leave the contradiction undocumented. |
| C-3 | **`workspace_members.role` exists with no behavioral effect.** The schema defines `role` (`'owner'` / `'member'`) per `database-schema.md`, but `architecture.md` §4 RLS policies treat every member identically — no policy or UI differentiates owner vs. member. This is the same kind of premature, unvalidated structure that `architecture.md` §2 explicitly rejected for the Organization layer ("don't build for an unvalidated need"). | **Medium** | Inconsistent application of the project's own stated principle; a future "only the owner can delete a workspace" feature will require both a migration-free policy rewrite *and* retrofitting UI — easy to forget since the column already silently exists today. | CTO decision: either drop `role` for Sprint 1 (every member is equivalent, add the column when ownership-gated actions are actually built) or keep it but explicitly document in `architecture.md` that it is inert in Sprint 1 and name the future feature it unblocks. |

## 2. Missing Requirements

| # | Issue | Severity | Impact | Recommendation |
|---|---|---|---|---|
| M-1 | **No edit or delete for workspace/project/task.** No document (`prd-sprint-1.md` User Stories, `technical-plan.md` routes/components) defines any update or delete operation — only creation and Kanban status moves. A typo in a task title or project name has no fix path. | **High** | First-time pilot users will hit this immediately (typos are inevitable in quick-add flows) and it directly undercuts trust and the "production ready" bar in `CLAUDE.md`'s Definition of Done. | Founder/PM: explicitly decide whether minimal rename/delete is in Sprint 1 scope or a documented known-gap for pilot, with a clear message to pilot users about how to report issues in the meantime. |
| M-2 | **No instrumentation/analytics plan to measure the metrics already committed to.** `prd-sprint-1.md` §7 and `executive-summary.md` §7 commit to measuring time-to-first-task, completion rates, and Kanban usage — but no document defines how these events are captured (no event logging, no timestamps beyond `created_at`/`updated_at` table columns). | **High** | The Go/No-Go decision in `executive-summary.md` §9 cannot be evaluated against real pilot data without this — the team would be flying blind on its own headline metric. | Product Engineer: add minimal event logging (sign-up, workspace_created, project_created, task_created, task_moved timestamps per user) to the technical plan before build starts. This instruments already-approved metrics; it does not add new product scope. |
| M-3 | **No password-reset / forgot-password flow.** Not mentioned in any document, despite being a near-certain pilot support request. | **Medium** | Locked-out pilot users become support escalations with no self-serve recovery path. | Founder/PM: confirm whether this is in Sprint 1 or an accepted gap covered by manual support during the small pilot. |
| M-4 | **No slug-generation/collision strategy.** `database-schema.md` defines `workspaces.slug` as `not null, unique`, used in routes (`technical-plan.md` §3), but no document specifies how a slug is derived from a workspace name or what happens when two workspaces would generate the same slug (e.g. two pilot orgs both named "Volunteers"). | **Medium** | First production collision will be a 500 error or a confusing UPSERT failure at workspace-creation time, directly inside the timed AC-6 flow. | Product Engineer: define slugification + a uniqueness-retry/suffix strategy in `technical-plan.md` before implementing workspace creation. |
| M-5 | **No rebalancing strategy for fractional `position`.** `database-schema.md` notes `position` (`numeric`) avoids re-indexing siblings on reorder, but no document defines an initial spacing value or a rebalancing trigger as values converge after repeated reorders (a well-known failure mode of fractional-indexing schemes). | **Medium** | Left unaddressed, heavy Kanban use in a single column can eventually hit floating-point/numeric precision limits, causing silent ordering bugs that are hard to reproduce later. | Product Engineer: define initial spacing (e.g. increments of 1000) and a periodic/triggered rebalance rule in `technical-plan.md` §7 before the `moveTask` action is built. |

## 4. Security Risks

| # | Issue | Severity | Impact | Recommendation |
|---|---|---|---|---|
| S-1 | **`workspace_members` INSERT policy is described by intent, not by enforceable rule.** `architecture.md` §4 says INSERT is "restricted to the creation flow," but if the actual RLS policy simply allows `INSERT ... WHERE user_id = auth.uid()` with any `workspace_id`, any authenticated user could self-insert membership into **any** workspace by calling the Supabase client directly (bypassing the intended `create_workspace_with_owner` RPC), instantly gaining read/write access to that workspace's projects and tasks. | **Critical** | Cross-workspace data exposure — directly violates the "Technical Success Criteria: RLS policies hold with no cross-workspace data leakage" already committed to in `executive-summary.md` §7. | CTO must specify, explicitly in `architecture.md` §4: **no client-facing INSERT policy on `workspace_members` at all.** The only INSERT path is the `create_workspace_with_owner` Postgres function running as `SECURITY DEFINER`, which bypasses RLS deliberately and is the sole writer. This must be stated as a hard rule, not left implicit. |
| S-2 | **No verification step for RLS before pilot release.** `executive-summary.md` §7 states RLS must hold with no cross-workspace leakage as a Technical Success Criterion, but no document defines how this will actually be tested (no RLS test plan, no cross-account manual or automated check). | **High** | A committed success criterion with no way to confirm it was met before real pilot data is at risk. | QA Lead: add an explicit RLS verification step (e.g. scripted test: user A cannot read/write user B's workspace data) to the development order in `technical-plan.md` §6, before pilot release (step 7 in `executive-summary.md` §8). |
| S-3 | **No secrets/environment management plan.** Supabase service role keys, anon keys, and any future secrets are not mentioned in any document. | **Low** | Low likelihood of an actual leak at this stage, but worth a one-line confirmation (e.g. Vercel environment variables, anon key only on the client) before first deploy. | Product Engineer: add a one-line note to `technical-plan.md` confirming key handling (anon key client-side only, service role key never shipped to the browser). |

## 5. UX Risks

| # | Issue | Severity | Impact | Recommendation |
|---|---|---|---|---|
| X-1 | **Drag-and-drop is the *only* way to change task status — no keyboard or screen-reader path.** `architecture.md` §6 and `ux-review.md` §6 both explicitly reject a status dropdown "to keep the interaction model to one mental model." This means a keyboard-only or screen-reader user has **no way at all** to move a task between columns. | **High** | This isn't just reduced affordance (already flagged in `ux-review.md` §8 for touch) — it's a complete accessibility blocker for an entire interaction class, and arguably contradicts `CLAUDE.md`'s "production ready" / "responsive design" bar. | UX Reviewer + Product Engineer: explicitly decide and document whether this is an accepted, time-boxed Sprint 1 limitation (with a named follow-up) or whether a minimal accessible fallback is required before pilot release. The current docs treat it as a settled design choice rather than a flagged risk — it should be the latter. |
| X-2 | **No defined mobile layout for the 4-column Kanban board.** `design-system.md`/`CLAUDE.md` require desktop-first **and mobile-friendly**, but `ux-review.md` never specifies how four side-by-side columns behave on a narrow viewport (horizontal scroll vs. stacked vs. single-column-at-a-time). | **Medium** | Without a decision, this gets improvised mid-implementation, risking an inconsistent or untested mobile experience right before a Definition-of-Done check ("works on mobile," "responsive design"). | UX Reviewer: add an explicit mobile Kanban behavior spec to `ux-review.md` §6 before the `KanbanBoard` component is built. |
| X-3 | **Single-workspace UX is implicitly assumed but not stated.** `ux-review.md` §3 mentions `WorkspaceSwitcher` → "New workspace," but doesn't address what the switcher looks like/does when a user has exactly one workspace (the expected case for nearly every Sprint 1 user, per C-2). | **Low** | Minor, but worth a one-line confirmation so the switcher isn't built as a dropdown with a single, pointless entry. | UX Reviewer: add a one-line note — switcher can be hidden or collapsed to a plain label when only one workspace exists. |

## 6. Technical Risks

| # | Issue | Severity | Impact | Recommendation |
|---|---|---|---|---|
| T-1 | **No automated test plan anywhere in the plan set.** None of the six documents mention unit, integration, or end-to-end tests, despite `CLAUDE.md`'s Definition of Done requiring "production ready" and the plan committing to specific, numbered Acceptance Criteria (`prd-sprint-1.md` §5) that imply verifiable behavior. | **High** | Acceptance Criteria with no test plan tend to be checked once manually and then drift as code changes; the ≤ 3-minute target and RLS guarantees are exactly the kind of thing that silently regresses without a test. | QA Lead: add a minimal test plan to `technical-plan.md` — at least one automated check per AC (AC-1 through AC-6) and the RLS cross-workspace check from S-2. |
| T-2 | **`create_workspace_with_owner` RPC privilege model is unstated.** `architecture.md` §5 names the function but doesn't specify it must run as `SECURITY DEFINER` to legitimately bypass RLS for the initial member insert (tied directly to S-1). | **High** | If implemented as a normal (non-`SECURITY DEFINER`) function, workspace creation will simply fail under RLS; if implemented carelessly, it could over-grant privileges. | CTO: add the `SECURITY DEFINER` requirement and its scope (only inserts into `workspaces` and `workspace_members`, nothing else) explicitly to `architecture.md` §5. |
| T-3 | **Cascade deletes are already baked in with no recovery path.** `database-schema.md` specifies `on delete cascade` from `workspaces` → `projects` → `tasks`. No delete feature is in scope yet (see M-1), but the schema-level behavior is already irreversible by design — deleting a workspace later (whenever that feature ships) permanently destroys all projects and tasks with no soft-delete or recovery window. | **Low** | Not an immediate Sprint 1 problem since no delete UI exists yet, but worth flagging now so the future delete feature isn't designed against an unreviewed assumption. | CTO: note in `database-schema.md` that cascade deletes are a deliberate MVP choice, to be revisited (soft-delete/trash) if/when a delete feature is scoped. |

---

## Summary for Go/No-Go

The most consequential items are **C-1** (email confirmation defeats the ≤ 3-minute target) and **S-1** (an under-specified RLS policy that could allow cross-workspace data access) — both are **Critical** and should be resolved in the documents themselves before implementation begins, not discovered during build. **C-2** (no invite path despite "whole team" framing) is a **High**-severity scope/messaging contradiction that leadership should resolve explicitly rather than let stand silently. The remaining items are real but lower-stakes — each has a concrete owner and a recommended decision above; none require new product scope to resolve, only clarification, correction, or an explicit accepted-risk sign-off within the existing plan.
